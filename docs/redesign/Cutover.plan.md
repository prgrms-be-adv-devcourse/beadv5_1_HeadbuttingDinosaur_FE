# Cutover 계획 (v2 → 메인 승격)

이 작업은 모든 v2 페이지 PR 머지 후 실행되는 마지막 단계.
v2 코드를 메인으로 승격하고 기존 코드를 일괄 삭제.

## 1. 사전 검증 (필수 체크리스트)

> Cutover PR 머지 **직전** 반드시 전부 통과해야 하며, 항목별로 “확인 방법”과 “통과 기준”을 PR 체크리스트에 링크한다.

1. **v2 라우트 동작 검증**
   - 점검 범위
     - 모든 v2 페이지를 `?v=2`로 접속하여 정상 동작 확인
     - 라이트 / 다크 모드 둘 다
     - 모바일 / 태블릿 / 데스크탑
     - 인증 / 비인증 상태별
     - 로딩 / 에러 / 빈 상태
   - 확인 방법
     - QA 시나리오 문서 기준으로 수동 테스트 실행
     - 주요 페이지는 Playwright/Cypress 스모크 테스트 병행
     - 브라우저 콘솔 에러/경고 확인
   - 통과 기준
     - P0/P1 기능 결함 0건
     - 콘솔 에러 0건(허용 예외 없음)
     - 상태별 UI가 명세와 일치

2. **v2 페이지 간 네비게이션**
   - 점검 범위
     - v2 페이지 간 이동 시 `?v=2` 유지
     - `Landing → EventList → EventDetail → Cart → 결제`
     - MyPage 탭 간 이동
   - 확인 방법
     - 라우트 전환 시 URL/쿼리 파라미터 캡처
     - 브라우저 뒤로가기/앞으로가기 포함해 탐색
   - 통과 기준
     - 지정 플로우 100% 성공
     - 중간 이탈/리다이렉트 오류 없음
     - `?v=2` 유실 케이스 0건

3. **v1 잔존 페이지 동작**
   - 점검 범위
     - INVENTORY § 6, § 7 범위 밖 페이지(admin, signup 등) 정상 동작
     - 새 토큰 적용 후 외관/레이아웃 깨짐 여부
   - 확인 방법
     - 범위 밖 페이지 목록 기반 스모크 테스트
     - 토큰 적용 전/후 시각 비교(스크린샷 diff 가능 시 수행)
   - 통과 기준
     - 핵심 액션(조회/저장/이동) 정상
     - UI 크리티컬 깨짐 0건

4. **기존 사용자 세션 / 데이터 호환**
   - 점검 범위
     - 로그인 상태 유지
     - 장바구니 데이터 유지(또는 명시적 마이그레이션)
     - 캐시 무효화 필요 시 명시
   - 확인 방법
     - 기존 브라우저 프로필(쿠키/스토리지 보유 상태)로 회귀 테스트
     - localStorage/sessionStorage key 변경 여부 확인
     - API 캐시 정책/버전 키 검토
   - 통과 기준
     - 재로그인 강제율 0%(정책적으로 허용한 경우 제외)
     - 장바구니 손실 0%(또는 사전 공지/마이그레이션 수행)
     - 캐시 무효화 전략 문서화 완료

5. **빌드 / 테스트**
   - 점검 범위
     - `tsc` 통과
     - `eslint` 통과
     - 단위/통합 테스트 통과
     - 번들 사이즈 비교(v2가 v1 대비 과도하게 증가하지 않는지)
   - 확인 방법
     - CI 파이프라인 및 로컬 동일 명령 실행
     - 번들 분석 리포트 생성(예: `analyze` 스크립트)
   - 통과 기준
     - 타입/린트/테스트 실패 0건
     - 번들 증가율이 팀 합의 임계값 이내(예: 초기 JS +15% 이하)

## 2. 영향 범위 매핑

> Cutover 시점 변경이 미치는 영역을 사전에 잠금(lock)하고, 각 영역 영향도를 `대/중/소`로 분류한다.

1. **라우트** — 영향도: **대**
   - 변경되는 라우트 목록(v1 → v2 전환)
   - 변경 안 되는 라우트(admin 등, INVENTORY 기준)
   - 산출물: “전환 라우트 목록표 / 비전환 라우트 목록표”

2. **디렉토리** — 영향도: **대**
   - 삭제: `src/pages/`, `src/components/` (전체 또는 일부)
   - 이동: `src/pages-v2/ → src/pages/`, `src/components-v2/ → src/components/`
   - 통합: `src/styles-v2/ → src/styles/`
   - 산출물: 파일 이동 매트릭스 + 충돌 목록

3. **라우터** — 영향도: **대**
   - Phase 0 `router-toggle` 헬퍼 제거
   - 분기 로직 단순화(v2 직접 매핑)
   - 산출물: 라우팅 진입점 변경 diff

4. **의존성(package.json)** — 영향도: **중**
   - v1 전용 라이브러리 제거 가능 여부
   - v2 신규 의존성 영구화 여부
   - 산출물: 제거/유지 의존성 표 + 근거

5. **CLAUDE.md** — 영향도: **중**
   - “v2 재구축 진행 중” 섹션 제거/갱신
   - `src/pages-v2/` 강제 규칙 등 임시 규칙 제거
   - 산출물: 운영 규칙 최신화

6. **테스트** — 영향도: **중**
   - v1 페이지 단위 테스트 삭제
   - v2 단위 테스트를 메인 경로로 이동
   - 산출물: 테스트 경로/coverage 변경 리포트

7. **CI / CD** — 영향도: **소~중**
   - 빌드 스크립트/lint 설정 영향 여부 확인
   - 산출물: 파이프라인 수정 필요성 체크(있으면 변경)

### 3.2 컴포넌트 삭제 대상

- `src/components/` 중 v2 대응 컴포넌트가 있는 항목 삭제
- 단, admin 등 v1 잔존 페이지에서만 사용하는 컴포넌트는 유지
- 삭제 전 “최종 사용처 0건”을 확인한 뒤 삭제

### 3.3 스타일 삭제 대상

- `src/styles/` 내 v1 전용 스타일(더 이상 import 없음) 삭제
- v2는 `src/styles-v2/` 기반이므로 cutover 시 통합 전/후 중복 확인 필수

### 3.4 기타 삭제 대상 (hooks/utils/contexts)

- v1에서만 사용하던 `hooks`, `utils`, `contexts` 식별 후 삭제 후보화
- 공용 유틸인지, v2 경유 사용인지 교차검증 필요

### 3.5 삭제 안전성 확인 방법

- import 추적(예시)
  - `rg "from ['\"]@/pages/" src`
  - `rg "from ['\"]@/components/" src`
  - `rg "from ['\"].*(pages|components|styles).*" src`
- 라우터/엔트리포인트 참조 확인
- 삭제 직후 `tsc`, `eslint`, 테스트로 정적/동적 검증

## 4. 이동 / rename 대상 (v2 → 메인)

### 4.1 이동 대상 표

| 현재 위치 | 이동 후 위치 | 비고 |
|---|---|---|
| `src/pages-v2/` | `src/pages/` | 폴더 통째 이동 |
| `src/components-v2/` | `src/components/` | 기존 파일 충돌 검토 필요 |
| `src/styles-v2/` | `src/styles/` | merge 또는 대체 전략 필요 |
| `src/lib-v2/` (있다면) | `src/lib/` | v2 공용 유틸 통합 |

### 4.2 이동 방법

1. **권장: `git mv`**
   - 장점: 히스토리 추적 용이
   - 단점: 대량 충돌 시 작업 복잡도 상승
2. **대안: `cp + rm`**
   - 장점: 작업 단순
   - 단점: 히스토리 단절

### 4.3 충돌 처리 방침 (`src/components`)

- 기존 `components/` 일부 유지 시 `components-v2/`와 파일명 충돌 가능
- 우선순위
  1) 기존 v1 컴포넌트 삭제 가능 여부 판단
  2) 삭제 가능하면 제거 후 v2 이동
  3) 삭제 불가하면 임시 네임스페이스(예: `legacy/`)로 분리

### 4.4 import 경로 변경 규칙

- `@/pages-v2/Login` → `@/pages/Login`
- `@/components-v2/Button` → `@/components/Button`
- `@/styles-v2/tokens.css` → `@/styles/tokens.css`

변경 파일 수 추정(명령 예시):
- `rg -n "@/pages-v2|@/components-v2|@/styles-v2" src`

### 4.5 자동화 가능성

- 1차 일괄 치환: `sed`(경로 치환)
- 2차 정리: IDE refactor / codemod(AST 기반)
- 3차 검증: `tsc` + `eslint` + 테스트

## 6. v1 ↔ v2 호환성 / 마이그레이션

### 6.1 인증 토큰

- 확인 항목
  - v1/v2 토큰 저장 형식 동일 여부
  - localStorage key 동일 여부
- 처리 방침
  - 동일: 마이그레이션 불필요
  - 상이: 1회성 마이그레이션 코드 추가 **또는** 강제 재로그인
- 권장
  - 사용자 영향 최소화를 위해 1회성 자동 마이그레이션 우선
- 코드 위치 / 제거 시점
  - 위치: 앱 부팅 초기화 레이어(auth bootstrap)
  - 제거: cutover 후 1~2주 안정화 확인 뒤 제거(PR 3)

### 6.2 장바구니 데이터

- 확인 항목
  - Cart plan § 3 결정(로컬/서버/하이브리드)과 일치 여부
  - v1 저장 포맷을 v2가 그대로 읽을 수 있는지
- 처리 방침
  - 포맷 호환: 유지
  - 포맷 비호환: 1회성 변환 또는 명시적 초기화
- 권장
  - 변환 실패 시 안전 fallback(빈 cart + 안내) 정의
- 코드 위치 / 제거 시점
  - 위치: cart hydrate 로직 직전
  - 제거: 변환 대상 비율 0% 확인 후 PR 3에서 제거

### 6.3 사용자 설정 / 테마

- 확인 항목
  - `data-theme` 속성 / localStorage key 동일성
  - Phase 0 ThemeContext 리팩터 후 영속화 방식 유지 여부
- 처리 방침
  - 키 동일: 유지
  - 키 변경: key alias 읽기(구키→신키 동기화)
- 코드 위치 / 제거 시점
  - 위치: theme provider 초기화
  - 제거: 구키 사용률 0에 수렴 후 제거

### 6.4 URL 쿼리스트링

- 확인 항목
  - v1 EventList 필터 URL이 v2에서도 유효한지
  - 키/값 체계 변화 여부(예: `cat=conference` 슬러그)
- 처리 방침
  - 호환 매핑 테이블 제공(구키/구값 → 신키/신값)
  - 비호환 파라미터는 무시 + 기본값 적용
- 코드 위치 / 제거 시점
  - 위치: 라우트 진입 파서/필터 파서
  - 제거: 레거시 링크 유입량 감소 후 제거

### 6.5 SSR / hydration (있다면)

- 확인 항목
  - 서버 렌더 페이지에서 v1↔v2 전환 시 깜빡임/불일치 여부
- 처리 방침
  - hydration mismatch 발생 시 분기 제거 전 렌더 경로 통일
- 코드 위치 / 제거 시점
  - 위치: SSR 엔트리/라우팅 경계
  - 제거: cutover 완료 직후 즉시 가능(분기 삭제)

## 8. 롤백 계획

### 8.1 즉각 롤백 (배포 직후 critical)

- 절차
  1) `git revert {cutover_pr_merge_commit}`
  2) hotfix 브랜치 배포
- 롤백 범위 명시
  - cutover PR 변경만 되돌림
  - 과거 v2 기능 PR 자체는 history에 남아있음

### 8.2 부분 롤백 (특정 페이지만 문제)

- 난점
  - cutover에서 v1 코드가 삭제되어 부분 롤백 복잡
- 옵션
  - a) 전체 revert 후 문제 페이지만 v2 재배포
  - b) 문제 페이지 임시 redirect/점검 페이지 적용
- 권장
  - 장애 강도 높으면 a), 낮으면 b)

### 8.3 데이터 롤백

- 핵심
  - §6 마이그레이션이 비파괴적이어야 롤백 용이
- 주의
  - 파괴적 변환(localStorage overwrite 등)은 롤백 어려움
- 원칙
  - 가능한 append/alias 방식의 비파괴 마이그레이션

### 8.4 모니터링 기반 의사결정

- cutover 머지 직후 **[N]시간 집중 모니터링**
- 관측 지표
  - 에러율, 핵심 퍼널 이탈, 사용자 CS, 서버 로그
- 임계값 초과 시 즉시 revert

### 8.5 revert 시뮬레이션 (머지 전)

- PR 체크리스트에 “revert 버튼 정상 동작 검증” 포함
- v1 코드가 git history에 온전하게 남아있는지 확인

### 8.6 배포 시점 권장

- 트래픽 적은 시간대
- 담당자(프론트/백/QA) 대기 가능 시간
- 다음날 휴일이 아닌 날

## 10. PR 분할 (골격만)

### 10.1 PR 1: 사전 정리 (선행 작업)

> 목표: Cutover 메인 PR 전에 독립 배포 가능한 작업을 분리하고, 기존 동작을 유지한다.

- 포함 파일 / 변경 사항
  - (필요 시) 데이터 마이그레이션 1회성 코드 추가
  - `src/components-v2/` ↔ 기존 `src/components/` 충돌 사전 해소(rename/namespace)
  - 완전 미사용 v1 코드 중 안전한 항목 선삭제(의존성 0)
- 추정 LOC
  - 중간 규모(예: `+200 ~ +800`, `-50 ~ -300`)
- 검증 방법
  - 회귀 테스트: “기존 동작 동일” 확인
  - `tsc` / `eslint` / 단위·통합 테스트
  - 마이그레이션 코드가 실제 데이터에 안전 적용되는지 샘플 검증
- 머지 후 모니터링
  - 최소 수시간~1일 관찰
  - 인증/장바구니/테마 마이그레이션 지표 확인

### 10.2 PR 2: Cutover (메인)

> 목표: v2를 기본 경로로 승격하고 v1 분기/코드를 제거한다. 이 PR 머지 시점부터 전 사용자 v2 노출.

- 포함 파일 / 변경 사항(대량)
  - v1 페이지/컴포넌트 삭제
  - v2 디렉토리 → 메인 디렉토리 이동
  - 라우터 토글 제거 및 라우트 직접 매핑
  - import 경로 전면 갱신
  - `CLAUDE.md`/환경 변수/설정 정리
- 추정 LOC
  - 대규모 예시: `-3000 / +500` (실측 필요)
- 파일 작업 순서(권장)
  1) 삭제 대상 정리
  2) `git mv` 기반 이동/rename
  3) import/라우터/설정 수정
  4) 테스트/문서 최종 정리
- commit 메시지 권장(작업 단위 분리)
  - `remove v1 pages`
  - `move v2 to main`
  - `update imports`
  - `remove router toggle`
- 검증 방법
  - 전체 라우트 스모크 + 핵심 사용자 플로우(E2E)
  - `tsc`, `eslint`, 테스트, 빌드, 번들 비교
- 의존 PR
  - PR 1 머지 및 안정 확인 후 진행

> 비고: PR 자체는 크게 보일 수 있으나, 중간 상태 깨짐 위험 때문에 기능 단위로 추가 분할은 제한적. 대신 commit 단위를 잘게 쪼개 추적 가능성 확보.

### 10.3 PR 3: 사후 정리 (후속 작업)

> 목표: Cutover 안정화 후 레거시 흔적 정리.

- 포함 파일 / 변경 사항
  - `docs/redesign/` 폴더 삭제(또는 일부 이관 후 정리)
  - PR1에서 추가한 마이그레이션 코드 제거
  - v1 전용 의존성(`package.json`) 제거
  - 임시 placeholder/TODO 제거
- 추정 LOC
  - 소~중 규모(예: `-300 ~ -1500`, `+0 ~ +100`)
- 머지 시점
  - cutover 후 **1~2주** 안정 운영 확인 뒤
- 검증
  - 레거시 참조(import/route/config) 잔존 0건
  - 제거된 항목 재의존 없음 확인

### 10.4 PR 간 의존성 / 머지 순서

```text
PR 1 (사전 정리) → 즉시 머지, 안정 확인
       ↓ 1~3일
PR 2 (Cutover 메인) → 머지, 집중 모니터링
       ↓ 1~2주
PR 3 (사후 정리) → 머지
```

#### 머지 시 검증 시나리오 매트릭스

| 시점 | 필수 검증 | 통과 기준 |
|---|---|---|
| PR1 머지 직후 | 기존 라우트/인증/장바구니 회귀 | 기존 기능 동일, 마이그레이션 오류 0 |
| PR2 머지 직후 | v2 전체 플로우 + 잔존 v1 페이지 | P0 결함 0, 에러율 임계값 이내 |
| PR3 머지 직후 | 문서/의존성/임시코드 제거 영향 | dead reference 0, CI 전체 통과 |

#### 배포 일정 권장

- PR2(cutover): **평일 저트래픽 시간대**(예: 화~목 오후 늦은 시간/야간 초입)
- 휴일 전날/인력 공백 시간대 배포 지양

#### 팀 커뮤니케이션

- 사전 공지: 머지/배포 일정, 롤백 담당자, 모니터링 채널 공유
- 머지 후 공지: 변경 요약, known issue, 대응 창구 공유
